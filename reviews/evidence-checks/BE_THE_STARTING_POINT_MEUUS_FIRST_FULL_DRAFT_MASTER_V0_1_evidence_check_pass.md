## Evidence-Check Principles

- Claim <= Evidence.
- Unknown must remain unknown.
- Future must be labeled future.
- Prototype must be labeled prototype.
- Draft must be labeled draft.

- Founder experience is not proof of system effectiveness.
- Manuscript language is not evidence.
- Public site state must be verified before public claim.
- Service availability must be evidenced before mention.
- Safety/trust claims require operational evidence.
- AI/DLAS/assessment claims require special review.
- Legal/therapy/religious/professional claims require qualified boundary language.

## Evidence Source Requirements

For each public-use claim category, required source types include:

- Public website state: screenshot, URL, deploy date, commit, release note.
- Prototype state: code path, demo note, limitation statement.
- Governance state: commit hash, tag/release, frozen document reference.
- Manuscript state: source record, version note, intake review.
- Research state: methodology file, dataset, participant evidence, analysis report, limitations.
- Legal/professional claims: qualified review or clear disclaimer.
- Therapy/recovery/mental health claims: qualified review or clear non-clinical boundary.
- Religious/spiritual claims: qualified review or reflection-only boundary.
- AI/DLAS claims: architecture evidence, validation status, runtime status, limitation statement.
- Payment/service claims: terms, pricing approval, delivery capacity, payment availability evidence.
- Impact claims: measured evidence, methodology, date range, limitation.
